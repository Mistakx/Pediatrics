    Desenvolvimento de Aplicação de Gestão

    Introdução

    Este trabalho consiste na implementação de uma Aplicação para Gestão (AG) de empresa cujo modelo é partilhado à frente  (disponível brevemente). A informação necessária para várias actividades de gestão é muitas vezes mantida em ficheiros Excel pois software comprado ou programado não é facilmente alterável ou consegue acompanhar de forma eficaz a mudança do negócio e dos requisitos. Vários processos de consolidação de informação requerem cálculos e operações manuais de copiar/colar, levando à inerente redundância de informação e possíveis erros nas operações devido ao fator humano. A AG a ser desenvolvida inclui vários componentes abaixo descritos que pretendem automatizar e tornar mais coerentes várias tarefas que manipulam informação importante.

    Preparação do ambiente de desenvolvimento

    A aplicação é para ser desenvolvida de forma integrada no software Wordpress, também usado como sistema gestor de conteúdos online (Content Management System). O Wordpress é uma aplicação desenvolvida sobre PHP+MySQL de forma a que todos os conteúdos de um website ficam armazenados numa BD, facilitando atualizações, tanto dos conteúdos como do próprio Wordpress. Um aspeto central do desenvolvimento do TDBD é a utilização do plugin Exec-PHP que permite que posts/páginas do Wordpress executem código PHP. Nomeadamente, os componentes da AG são para ser desenvolvidos em páginas gravadas numa instalação do Wordpress as quais irão conter referências a ficheiros contendo todo o código PHP e SQL necessário. As grandes vantagens desta abordagem é a rapidez e facilidade com que se organizam os vários componentes/subcomponentes, bem como a utilização de plugins/mecanismos/funcionalidades do próprio Wordpress para aumentar as capacidades da AG e também a realização de backups/restores.

    Os alunos terão de instalar um hypervisor que possa correr uma máquina virtual que irá conter o sistema operativo Linux (distribuição Debian), incluindo o servidor web Apache com PHP, o SGBD MySQL e o Wordpress já instalados. Nomeadamente, será necessário realizar download desta Máquina virtual (link alternativo no onedrive da UMa) - compatível com vários hypervisors como: VirtualBox (Windows, Mac e Linux, clicar no link para o respetivo sistema operativo, versão mais recente), Gnome Boxes (Linux) e VMware Workstation Player (Windows e Linux), entre outros - e importá-la. Para correr máquinas virtuais poderá ser necessário ativar a opção de virtualização do CPU na BIOS do respetivo PC, no caso de Intel chama-se VT-x. É preciso entrar na BIOS antes do arranque, encontrar essa configuração (pode ser necessário ativar mostrar configurações avançadas) e ativar.

    Após o arranque do sistema na máquina virtual, a consola irá mostrar as mensagens:

    *** You can access the application at http://192.168.134.112 ***
    *** The default username and password is 'user' and 'asdfghjklzxc' ***

    Tomar nota do conjunto de 4 números separados por 3 pontos (exemplo a vermelho acima) que consiste no endereço IP em que o servidor virtual está acessível. Este endereço IP é daqui para a frente referido como serverip sendo que, no restante enunciado, sempre que aparece serverip é para substituir por este endereço IP atribuído à máquina virtual.

    Tomar nota também da password gerada automaticamente (exemplo a azul acima) a usar nas aplicações wordpress e phpmyadmin.

    Realizar login na consola com o user e password bitnami e, logo de seguida, será solicitada a mudança de password que terá de ser: sgbd#XNN em que X é letra do código de grupo e NN o número do código de grupo (por exemplo, para o grupo A01 a password será sgbd#A01). Introduzir a password atual (bitnami) e depois a nova: sgbd#XNN.

    Depois, executar os seguintes comandos na consola:

    sudo rm -f /etc/ssh/sshd_not_to_be_run
    sudo vi /etc/ssh/sshd_config

    Isto irá abrir o ficheiro de configuração num editor de texto e é para carregar na tecla i para ativar modo INSERT, ir até ao fim do ficheiro e mudar PasswordAuthentication de no para yes sendo depois necessário carregar na tecla Esc para sair do modo de edição, seguido de :wq e Enter para gravar e sair (nota para obter o carater : poderá ser necessário usar a tecla ç de cedilha). Depois, executar os seguintes comandos na consola:

    sudo systemctl enable ssh
    sudo systemctl start ssh

    E, desta forma, já será possível:

    • aceder ao disco da máquina virtual, usando-se para isso qualquer programa que permita aceder a um servidor por ssh ou "montar" uma drive virtual no sistema operativo host com uma ligação ssh usando-se como dados de login bitnami e password sgbd#XNN. Por exemplo, em Windows recomenda-se SFTP Drive (NOTA: no windows ao ser montada a drive virtual, irá mostrar os conteúdos da pasta /home/bitnami e para aceder à pasta raiz do wordpress como indicado abaixo deverá clicar-se na letra da drive montada para ir à raiz da drive da máquina virtual). Em Linux pode-se usar o próprio explorador do Gnome (o nautilus) e escolher opção "Other Locations", colocar sftp://serverip e criar um bookmark para mais rápido acesso no futuro.
    • executar comandos no sistema Linux da máquina virtual, com um terminal cliente SSH, em vez de através da consola, o que é mais prático para copiar/colar os comandos indicados ao longo do enunciado, pois nem todos os hypervisors permitem colar texto na consola da máquina virtual e pode haver problemas em manter a configuração do teclado; NOTA: no resto do enunciado, sempre que se indica execução de comandos na consola, podem ser executados através do terminal cliente SSH na máquina host ou diretamente na consola dentro da máquina virtual; para o Windows recomenda-se o Putty e para Linux/Mac, basta usar o terminal de comandos e executar o comando (password é sgbd#XNN):
      • ssh serverip -l bitnami

    NOTAS:

    • Se não estiver a ser possível o acesso à máquina, é provável que hajam problemas com o modo de rede e tenha de mudar-se o mesmo nas configurações da máquina para host-only, como indicado no enunciado abaixo. Isto implica que será necessário fazer download dos plugins do wordpress abaixo listados manualmente e instalá-los da mesma forma que o php-exec.
    • A máquina virtual não deve ser desligada com um "power off" ou simplesmente fechando o software hypervisor, pois corre-se o risco de a máquina ficar corrompida, especialmente o SGBD mariadb; para desligar a máquina ou reiniciá-la devem ser utilizados na consola, respetivamente os comandos sudo shutdown ou sudo reboot


    De seguida deverá ser executado o seguinte comando na consola:

    sudo vi /opt/bitnami/apache/conf/bitnami/phpmyadmin.conf

    Isto irá abrir o ficheiro de configuração num editor de texto e é para carregar na tecla i para ativar modo INSERT e editar o ficheiro de modo a ficar o seguinte conteúdo (estando a negrito o realmente importante; as linhas começando com # podem permanecer pois o # funciona para tornar a linha como "comentário"):

    Alias /phpmyadmin "/opt/bitnami/phpmyadmin"
    <Directory "/opt/bitnami/phpmyadmin">
     Options -Indexes +FollowSymLinks -MultiViews
     AllowOverride All
     Order Allow,Deny
     Allow from all
     Require all granted
    </Directory>

    sendo depois necessário carregar na tecla Esc para sair do modo de edição, seguido de :wq e Enter para gravar e sair

    Depois executar os seguintes comandos:

    cd /opt/bitnami
    sudo ./ctlscript.sh restart apache

    Neste momento o phpmyadmin já será acessível da máquina host pelo seguinte endereço:

    http://serverip/phpmyadmin

    NOTA1: os dados de acesso são: user root e a password é a que foi gerada automaticamente e aparece após o boot na consola (exemplo a azul em cima).

    NOTA2: caso não esteja a ser possível conexão ao sgbd mariadb, pode ser que:

    • o mesmo não esteja a correr ou esteja a correr num IP/porta errado e pode ser resolvida a situação com um restart com o seguinte comando: ./ctlscript.sh restart mariadb
    • tenha havido corrupção da base de dados interna; este link tem várias instruções para se resolver este problema. ATENÇÃO: na página indicada, o comando mysqld deve ser precedido de sudo

    ATENÇÃO: em caso de problemas na conexão à máquina virtual, tentar as soluções indicadas abaixo na secção: NOTAS IMPORTANTES

    Depois aceder a: http://serverip/wp-admin para se iniciar a configuração básica do próprio Wordpress.

    NOTA: se o wordpress não estiver acessível, pode bastar um restart do apache e/ou da máquina virtual.

    O Username é user e a Password a indicada na consola gerada automaticamente. Levar o cursor até ao canto superior direito onde tem o ícone de utilizador e depois selecionar a opção Edit my profile seleccionar a opção Disable the visual editor when writing (no topo da página) e clicar no botão Update profile, no fim da página.

    Ir à opção do menu Plugins e escolher Add New. Procurar, instalar e ativar os seguintes plugins (para cada, usar botão Add New):

    1. Classic Editor (para usar editor clássico do Wordpress)
    2. PublishPress Capabilities (a ser usado mais tarde)
    3. Exclude Pages From Menu (para excluir páginas do público geral)
    4. Simple Page Ordering (para facilmente ordenar páginas criadas)
    5. Don't Muck My Markup (desativa auto-formatação do Wordpress, necessário para se poder controlar completamente o output HTML realizado pelo Wordpress)

    Instalar manualmente o plugin Exec-PHP disponível neste ficheiro zip, descompactando e copiando a pasta resultante exec-php para a pasta /opt/bitnami/wordpress/wp-content/plugins na instalação do wordpress na máquina virtual.

    Executar o seguinte comando na consola:

    sudo chown -R bitnami.daemon /opt/bitnami/wordpress/

      Clicando na opção Pages no Dashboard do Wordpress é possível começar a criar (clicar em "Add New") páginas para desenvolver os componentes abaixo especificados. Por exemplo, ao criar-se (opção Publish) a página com título Gestão de registos. É automaticamente criado o "permalink" http://serverip/gestao-de-registos onde fica acessível o respectivo componente (nota: mandar para o lixo a página Sample). Criar as pastas custom/php na pasta raiz do wordpress (i.e., /opt/bitnami/wordpress/custom/php e, para cada componente (por exemplo o gestao-de-registos):

      • criar uma página no Wordpress com o nome do componente (opção "Add New") e escrever título como especificado no enunciado, por exemplo: Gestão de registos
      • colocar como conteúdo: <?php include("custom/php/gestao-de-registos.php"); ?>, carregar no botão azul Publish à direita.
      • criar o ficheiro gestao-de-registos.php na pasta custom/php
      • realizar a programação directamente no ficheiro php criado, colocar por exemplo: <?php echo "<h1>HELLO WORLD</h1>"; ?> no ficheiro php e aceder à página Gestão de registos no frontend do Wordpress para verificar se o código php está sendo executado.

      NOTAS IMPORTANTES

      • Para editar os ficheiros php, usar um editor que permita gravar os ficheiros com codificação utf8 e usar sempre esta codificação, senão poderão ocorrer erros estranhos na execução.
        • Recomenda-se especialmente o phpstorm (versão gratuita para estudantes aqui) dada a sua versatilidade a nível de auto-complete, syntax highlighting e outras. Em alternativa recomenda-se também o Notepad++.
      • Caso o código php não esteja a ser executado, ou hajam problemas de permissões, ou o erro "Internal Server Error" no acesso às páginas, executar o seguinte na consola deverá resolver o problema (usar seta para cima do teclado para aceder a comandos executados recentemente na consola):
        • sudo chown -R bitnami:daemon /opt/bitnami/wordpress
      • Para evitar problemas de conexão, deve-se configurar a máquina virtual para usar NAT e não bridged (que pode dar problemas com atribuição de IPs)
      • Pode não ser possível aceder à máquina virtual ao mudar-se de rede (por exemplo de casa para UMa, de ligação de dados para wireless ou vice-versa, de wifi para cabo...)
        • Poderá ser possível ter acesso à máquina configurando a rede na máquina virtual como bridged ou só com NAT ou só com ligação de dados ou só em casa por exemplo.
        • Como uma vez instalados o teclado pt, plugins e tema na máquina virtual, já não é preciso esta aceder à Internet, uma solução que em princípio funciona para todos os casos é configurar a rede na máquina virtual no modo host only e assim o acesso à máquina virtual já irá funcionar sem conflitos. Neste modo o Edge da Microsoft pode não conseguir aceder ao IP da máquina virtual, mas o Chrome sim, então testar com diferentes browsers. Uma forma de confirmar que a máquina virtual já está ligada e acessível é executar o comando ping xxx.yyy.ccc.ddd (substituir letras pelo IP da máquina virtual) ou tentar ligar ao servidor por ssh ou sftp.
        • Se persistirem problemas de conexão na virtualbox, algo que poderá resolver é a geração de um novo mac address nas configurações de rede
      • Caso haja perda de acesso ao wordpress mas a conexão com a máquina virtual funciona para sftp ou phpmyadmin, mesmo mudando o modo de rede, será devido a mudança de IP ou necessidade de reiniciar a máquina virtual, sendo que terá sido atribuído um novo ip à máquina. Normalmente reiniciando (comando na consola: sudo reboot) e depois executando o comando sudo /opt/bitnami/configure_app_domain --domain serverip (novo IP atribuído à máquina virtual) resolve-se o problema de acesso ao wordpress.

      Alternativa à Máquina Virtual

      Poidem acontecer alguns (raros) casos de, mesmo em modo host-only e testadas as várias alternativas acima, a máquina virtual ficar inacessível.

      Para quem usa Windows, em alternativa a usar máquina virtual, é possível a instalação da solução da Microsoft WSL, que permite correr várias distribuições de Linux "diretamente" no Windows sem necessidade de máquina virtual. Após a instalação do WSL, deverá ser instalada a distribuição debian e depois pode ser instalada dentro do sistema debian a stack bitnami wordpress para linux que instala todo o software necessário e presente na máquina virtual bitnami (apache, mariadb, phpmyadmin, wordpress, etc.).

      Quem esteja a usar Linux ou Mac OS X como sistema operativo principal pode, em alternativa a usar máquina virtual, também instalar diretamente a respetiva stack bitnami.

      Não é permitido a utilização da stack bitnami para Windows.

      Como vantagens desta solução alternativa da stack, deixa de ser necessário o recurso a SFTP/SSH para controlar os serviços e ficheiros. Quem esteja com problemas recorrentes na máquina virtual pode também "migrar" para esta alternativa.

        Esquema relacional da BD da AG

        O esquema está disponível num ficheiro MWB do MySQL Workbench (MSWB) aqui (consultar com regularidade o enunciado/fórum pois poderão ser publicadas novas versões): v01, v02, v03. Os ficheiros SQL gerados a partir dos esquemas e prontos a importar através do phpMyAdmin estão aqui: v01, v02, v03. Scripts de alteração com os comandos necessários para "upgrade" de cada versão: v01-> v02 e v02-> v03. Todo o código SQL disponibilizado neste enunciado é para ser executado na Base de Dados bitnami_wordpress acessível no phpmyadmin, que contém as tabelas pré-definidas do wordpress e irá conter as tabelas da AG.

        Este esquema é para ser escrupulosamente respeitado sendo que todos os grupos não poderão realizar alterações ao mesmo. Este trabalho tem como foco o desenvolvimento de scripts PHP para concretizar os componentes pretendidos, contendo formulários e instruções SQL para inserção e visualização dos dados relevantes. No entanto, caso seja, por parte dos alunos, detetada alguma falha ou inconsistência no esquema definido, ou observada uma possível melhoria do esquema para torná-lo mais eficaz ou flexível, deve ser colocado um post no fórum do TDBD, para que a sugestão possa ser avaliada, podendo levar a eventual alteração do esquema base.

        Clicando nas tabelas no MSWB, é possível visualizar eventuais comentários criados para a tabela e para os atributos, que explicam a razão de ser da tabela ou do atributo e como se relacionam com outras tabelas e/ou atributos. Tabelas e/ou atributos óbvios (como ids) não têm comentários. A leitura dos comentários é importante para o esquema relacional ser bem interiorizado e percebida a lógica de relacionamento das entidades em que os vários componentes da AG irão basear-se o que, por sua vez, é essencial para que o desenvolvimento da aplicação seja intuitiva e eficaz.

        Deverão ser inseridos tuplos suficientes nas tabelas que permitam o teste com sucesso dos componentes associados, podem e devem ser aproveitados e expandidos os exemplos constantes na especificação dos componentes abaixo.

        Segue-se um conjunto de regras e dicas a seguir na execução do trabalho e, após esta secção, o conjunto de requisitos e funcionalidades a implementar em cada componente. Para cada componente é especificado o comportamento esperado da AG e dadas instruções ou dicas para a sua implementação em PHP+SQL.

        Regras/dicas a seguir

        Configurações adicionais no Wordpress

        • Deve ser ativada a flag de DEBUG no ficheiro de configuração do wordpress para serem mostrados eventuais erros no código PHP - no ficheiro wp-config.php colocar:
          define('WP_DEBUG', true); usar por exemplo o comando vi /opt/bitnami/wordpress/wp-config.php no terminal
        • Terão de ser criadas, na pasta custom, duas pastas com os nomes css e js que irão conter os ficheiros CSS e Javascript necessários (por ex. os ficheiros .js das bibliotecas de validação opcionais).
        • Executar o seguinte comando na consola:
          • sudo chmod 664 /opt/bitnami/wordpress/wp-content/themes/responsive/functions.php
        • Para usar-se CSS, os grupos terão de colocar (conforme esta fonte) o seguinte no final do ficheiro functions.php do theme (Ir a: Appearance->Theme Editor, clicar I Understand no aviso e, no lado direito, procurar por Theme functions e clicar no respetivo link):
          • wp_enqueue_style('ag', get_bloginfo( 'wpurl' ) . '/custom/css/ag.css',false,'1.1','all');
          • Nota: na pasta css, tem de ser criado um ficheiro com o nome ag.css onde podem ser colocadas as instruções CSS a serem aplicadas nos conteúdos HTML dos componentes da AG. Todos os estilos CSS utilizados pelos grupos (inclusivé os das bibliotecas de validação) têm de ser copiados para dentro deste ficheiro
        • Para implementar-se a funcionalidade (opcional) de validação client-side com Javascript recomenda-se (seguindo mesma fonte acima) código do género:
          • wp_enqueue_script( 'script', get_bloginfo( 'wpurl' ) . '/custom/js/script.js', array ( 'jquery' ), 1.1, true);
        • Para utilização de variáveis de sessão, é necessário (e obrigatório) incluir no fim do ficheiro functions.php (conforme esta fonte):
          • function register_my_session() {
                if (session_status() == PHP_SESSION_NONE) {
                    session_start();
                }
            }
            add_action('init', 'register_my_session');
          • Depois para criar/utilizar variáveis de sessão é só usar algo do género: $_SESSION['nome_da_var']=$_REQUEST['atributo'];
          • Caso hajam problemas na utilização de variáveis de sessão, tentar o seguinte para detectar o erro:
            • echo da variável de sessão antes e depois de a utilizar
            • certificar que o nome da variável de sessão corresponde e é igual: no momento da sua criação e no momento da sua utilização
            • certificar que NÃO está ser feito o unset da variável sessão no código antes de ser utilizada

          Regras a seguir na programação

            • Criar um ficheiro chamado common.php que terá de ficar na pasta custom/php.
              • Este ficheiro terá de incluir todas as funções PHP que são para ser reutilizadas entre componentes diferentes e/ou chamadas a bibliotecas Javascript (por exemplo, funções de validação server-side, chamada à bibliotecas, etc.). NOTA: Código a ser reutilizado em apenas um componente tem de ser colocado em funções definidas no próprio componente.
            • Nem sempre está a ligação ativa por parte do wordpress à BD, pelo que, pelo seguro, convém sempre realizar ligação sempre que necessário, o que terá de ser feito com código do género (a linha pode ser colocada numa função no ficheiro common.php):
              $link = mysqli_connect(DB_HOST,DB_USER,DB_PASSWORD,DB_NAME);
              em que na 1ª linha estão constantes definidas automaticamente pelo Wordpress devendo o código ficar exatamente como está. O uso deste código é obrigatório por parte de todos os alunos, senão o código não irá funcionar em máquinas virtuais dos colegas ou docentes aquando dos testes. Para depois executar queries basta usar código do género: 
              $result = mysqli_query($link,$query);
            • Não podem ser criadas referências/links a caminhos absolutos contendo o IP da máquina e/ou nome da página/componente e, (nos formulários) ou se tem de usar o atributo action vazio, ou têm de ser usadas funções do wordpress para se criar dinamicamente os links necessários. Exemplos:

              • na especificação de um action de um formulário (se não funcionar action="") ou de um link (por ex. [editar]), em vez de um link do género: http://10.42.43.10/wordpress/gestao-de-registos tem de se usar código do género:

                • no common.php:
                  • global $current_page; $current_page = get_site_url().'/'.basename(get_permalink());

                • e, no ficheiro php do respetivo componente:

                • echo '<form method="post" action="'.$current_page.'">';
                • echo '<a href="'.$current_page.'?estado=editar&item='.$item_id.'">[editar]</a>';
            • em relação ao ficheiro common.php, explicado acima, todos os ficheiros php de componentes (como o gestao-de-registos.php) terão de ter como primeira instrução PHP (logo a seguir a <?php) a seguinte (inclusão com caminho relativo de código PHP de um ficheiro): require_once("custom/php/common.php");

          Dicas a seguir na programação

          • Atenção às maiúsculas e minúsculas! No Linux os nomes de ficheiros e nomes de tabelas/atributos em SQL são case-sensitive. Os nomes dos ficheiros têm de ter letras todas minúsculas e os nomes das tabelas/atributos é também para ficar tudo em minúsculas.
          • Não se recomenda intercalar blocos de código php (com as tags <?php e ?>) com blocos de código HTML pois pode levar a problemas de sintaxe relativamente a chavetas php, funções e outros problemas. Recomenda-se um ficheiro php começar e terminar sempre com as tags <?php e ?> e, quando for necessário realizar output de um bloco HTML, fazê-lo com um echo com concatenações conforme necessário (ver exemplo do action do form acima) ou um echo de várias linhas de html, por exemplo:
            • echo "<table>
                  <tr>
                      <th>id</th>
                      <th>name</th>";
          • Atenção que nos formulários HTML não devem haver campos com nome "name" pois poderá dar problemas na execução, usar nomes do género person_name, etc.
          • Na especificação dos componentes (apresentada na secção do enunciado após as dicas) é indicado, para os formulários a constar nos mesmos, para cada campo, o seguinte conjunto de itens separados por um traço:
            • a negrito - tipo de input - em que o nome do respetivo campo deverá ser uma string sempre em minúsculas e com underscore em vez dos espaços, devendo tal nome ser idêntico ou próximo do nome do respetivo atributo existente na BD, caso apareça um nome entre parêntesis à frente do tipo de input, é esse o nome para ser usado como nome do campo e que irá corresponder ao nome da chave no array POST/REQUEST
            • a itálico - descrição do campo que tem de aparecer no output antes do mesmo
            • dentro de parêntesis - regras de preenchimento do campo (que deverão ser mostradas também no output junto aos respectivos campo da forma mais intuitiva, normalmente após os mesmos, à direita ou abaixo)
            • sublinhado - dicas/instruções de apresentação

          • Todos os comandos php mysql deverão incluir a letra "i" (como os acima) pois correspondem à versão mais recente e a sem o "i" nem sempre funciona como esperado, estando em fase de ser deprecada.

          • O uso de "required" na tag HTML input não tem comportamento garantido pois browsers antigos não suportam esta funcionalidade, pelo que não é permitido o uso desta funcionalidade e só pode ser feita validação server side (parte dos mínimos) e validação Javascript client-side que deve ser facilmente desativada com o uso de uma variável de configuração no ficheiro common.php, testada sempre que é para fazer output de algum script de validação.

NOTA IMPORTANTE: a utilização de transacções não é necessária para obtenção de nota mínima (embora seja atribuído um certo peso na nota final), logo deixar para mais tarde em caso de dificuldades.

                            Em todas as tabelas dos componentes anteriores que tenham os elementos [editar], [ativar] e [desativar], estes deverão ser ligações em que:

                            • no 2º e 3º casos, é um link para esta página e, pelo método GET a variável estado vem com o valor ativar ou desativar, conforme o caso e vem outras duas variáveis (escolher o nome conforme o componente) indicando o tipo e o id do elemento a ativar ou desativar (mudar o valor do atributo state na respetiva tabela); nos respetivos estados, deverá ser apresentada a pergunta se o utilizador deseja ativar ou desativar o elemento e um botão de input num formulário que, ao ser clicado, é mudado o valor do atributo state do respetivo tuplo da respetiva tabela

                            O componente gestão de registos, ao apresentar valores dos subitens de cada item, o nome do item deverá ser um link para esta página, em que pelo método GET, a variável estado vem com valor editar e vem uma variável com nome item com valor igual ao id do qual deverão ser editados os valores e outra com nome crianca com valor igual ao id da criança cujos valores irão ser editados; nesta situação deverá ser apresentado formulário idêntico ao do componente inserir valores (pode e deve ser reaproveitado o código existente) mas apresentando-se os campos já pré-preenchidos/selecionados com os respetivos valores constando na tabela value.
